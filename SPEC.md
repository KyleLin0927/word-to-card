# Word-to-Card Project Spec

> **[Meta-Rule: Documentation First]**
> 當需求變更或技術棧調整時，必須先更新此文件並取得確認，才可修改程式碼。

## 專案目標
英文單字卡搜集工具。
macOS 全域工具：
- 快捷鍵截圖 -> Gemini Vision 辨識 -> Anki 自動製單字卡
- 反白 + 快捷鍵 -> 剪貼簿攔截 -> Gemini Text 生成 -> Anki 自動製單字卡
具備錯誤容忍機制與自動清理功能。

## 技術棧
- **Language**: Python 3.10+ (環境隔離於 `.venv`)
- **AI Model**: Google Gemini（預設 `gemini-2.5-flash`，可透過 `.env` 的 `GEMINI_MODEL` 覆寫；若指定模型不可用需自動選擇可用模型）
- **Integration**: AnkiConnect (localhost:8765)，下載的代號是 2055492159
- **Dependencies**: `requests`, `python-dotenv`, `pynput`, `Pillow`, `pyperclip`, `edge-tts`

## 核心檔案結構與功能
1. **`config.py`**: 
   - 讀取 `.env` 中的 `GEMINI_API_KEY`。
   - 設定 Anki 牌組名稱與模型名稱（預設 `Basic`，可透過 `.env` 的 `ANKI_MODEL_NAME` 覆寫）。
2. **`notify.py`**: 
   - 封裝 `osascript` 提供系統彈窗通知。
   - 標題與內文在送交 AppleScript 前須**安全化**（例如將換行改為空白、轉義 `\\` 與 `"`、過長截斷），避免因字元破壞 `osascript` 導致通知無聲失敗；若 `osascript` 回傳非零結束碼應寫入 log 以利除錯。
3. **`queue_manager.py`**: 
   - 離線任務佇列：若 Gemini 或 Anki 連線失敗，將請求參數存入 `pending_tasks.json`。
   - 提供 `process_queue()` 於網路恢復時重試。
4. **`history_logger.py`**: 
   - 記錄已成功的單字卡，防止重複建立。
5. **`word_archive.py`**:
   - 在 Anki **實際成功新增**的單字上，將 Gemini 產生的完整單字 JSON 另存至本地目錄（與 Anki 牌組並行，供使用者自行備份或版本管理）。
6. **`main.py`**:
   - 監聽快捷鍵：
     - 截圖模式：`<cmd>+<ctrl>+s`
     - 反白取詞模式（剪貼簿攔截）：預設 `Ctrl+C`（可透過 `.env` 覆寫）
   - 觸發 `screencapture -i` 存至 `captures/` 目錄。
   - 整合 Gemini Vision API。
   - 反白取詞模式：模擬 `Cmd+C` 將反白文字送入剪貼簿 → 讀取文字 → 還原剪貼簿 → 呼叫 Gemini 產生單字卡 JSON。
   - **Cleanup**: 成功匯入 Anki 後，自動刪除 `captures/` 中對應的圖片檔案。
   - **本地單字庫**：成功寫入 Anki 後，同步寫入 `word_archive`（見下方需求）。

## Gemini Vision 指令規範 (Prompting)
- 角色：專業英文老師。
- 任務：找出截圖中心最核心的英文生字。
- 優先順序：**優先辨識位於截圖畫面中心位置的單字；若中心位置不明顯，則挑選畫面中最符合 GRE/TOEFL 難度的單字。**
- 視覺提示：若圖片模糊、無文字或無法辨識，請回傳 `{"error": "IMAGE_UNCLEAR"}`。
- 正常輸出：嚴格 JSON 格式 (word, meaning, example)。
- 例句要求：需符合 GRE/TOEFL 學術難度，且需提供 **2 句**（用同一個欄位承載，建議以換行 `\n` 分隔兩句）。
- 常用用法要求：需提供常見搭配（特別是介系詞搭配），例如 `interested in`, `capable of`, `apply for`。

## Gemini Text 指令規範（反白取詞模式）
- 角色：專業英文老師（GRE/TOEFL）。
- 輸入：使用者反白並複製到剪貼簿的文字（通常是一個單字，可能包含標點或前後空白）。
- 任務：若輸入為單字（或可合理正規化為單字），產生 1 筆單字卡資料。
- 正常輸出：嚴格 JSON 物件（同單字卡欄位結構：`word`, `phonetic`, `part_of_speech`, `definition`, `definition_zh`, `difficulty`, `example_sentence`, `synonyms`）。
- 例句要求：需符合 GRE/TOEFL 學術難度，且需提供 **2 句**（用同一個欄位承載，建議以換行 `\n` 分隔兩句）。
- 常用用法要求：需提供常見搭配（特別是介系詞搭配），例如 `interested in`, `capable of`, `apply for`。
- 異常輸出：
  - 若輸入不是單字（例如空字串、過長句子、或無法判斷），請回傳：`{"error":"NOT_A_WORD"}`。
- 只回傳 JSON，不要任何說明文字或 markdown。

## 需求：反白取詞（剪貼簿攔截）作為第二種輸入方式
### 背景
- macOS 沙盒/權限限制下，Python 無法直接取得其他應用程式目前反白的文字。
- 實務採用「剪貼簿攔截法（Clipboard Interception）」：用程式模擬 `Cmd+C`，再從剪貼簿取值。

### 流程
- **觸發**：使用者反白文字後按 `Ctrl+C`。
- **模擬拷貝**：程式收到快捷鍵後，立即模擬按下 `Cmd+C`。
- **讀取與還原**：
  - 先保存原剪貼簿內容
  - 讀取新的剪貼簿文字
  - 完成後還原剪貼簿內容（避免影響使用者）
- **送出 API**：把文字丟給 Gemini Text 產生單字卡 JSON。
- **寫入 Anki**：與截圖模式共用同一套去重/歷史/Anki 寫入規則。

### 非功能需求
- **安全性/權限提示**：若因 macOS 權限導致無法攔截快捷鍵/模擬按鍵，需在終端與通知中提示使用者到「系統設定 → 隱私權與安全性 → 輔助使用」開啟權限。
- **延遲控制**：模擬 `Cmd+C` 後需等待極短時間（例如 50–150ms）再讀剪貼簿，降低讀到舊值的機率。
- **監聽相容性**：熱鍵監聽需採用與目前 `pynput` 版本相容的實作，避免 `GlobalHotKeys._on_press()` 參數簽名不一致造成 listener callback crash。

### 驗收條件
- **AC1**：反白任意單字後按 `Ctrl+C`，程式能成功取得該字並呼叫 Gemini Text 回傳 JSON，最後新增到 Anki。
- **AC2**：處理完成後剪貼簿內容需被還原（使用者原本剪貼簿內容不應被覆蓋）。
- **AC3**：若反白內容為空或非單字，程式不得新增卡片，並以通知告知失敗原因（例如 NOT_A_WORD）。
- **AC4**：此模式也必須遵循「單字去重」規則（同批/歷史/Anki duplicate 對齊）。

## 需求：Gemini 模型可用性與降級策略
- **可配置模型**：允許透過環境變數/`.env` 的 `GEMINI_MODEL` 指定模型名稱。
- **自動降級**：當 `GEMINI_MODEL` 指定的模型發生「不存在 / 不支援 generateContent」等錯誤時，程式需自動改用可用且支援 `generateContent` 的 Gemini 模型重試。
- **可觀測性**：需要在終端輸出實際使用的模型名稱（若發生降級，需輸出降級前後的模型）。

### 驗收條件
- **AC1**：執行 `python main.py --test` 時，若 `GEMINI_MODEL` 指向不存在的模型，不得直接因 404/NOT_FOUND 中止；需自動選擇可用模型並完成一次 `generate_content` 呼叫（除非 API key/網路本身不可用）。
- **AC2**：`GEMINI_MODEL` 指向可用模型時，不應額外列舉模型或改用其他模型。

## 需求：Gemini 回傳格式防禦
- 程式需能容忍 Gemini 偶發的輸出格式偏差（例如回傳單一物件、`{"words":[...]}` 包裝、或含 markdown code fence）。
- 在進入主流程前，需將回傳正規化為 `list[dict]`，且每筆至少包含有效 `word` 欄位。
- 若輸出不符合可解析格式，流程不應造成 thread crash，需轉為一般失敗處理（通知/佇列）。
- **可觀測性**：當 Gemini 回傳內容無法 JSON 解析時，log 需記錄「原始回應內容」與「清理 code fence 後內容」（可截斷）以利除錯。

### 驗收條件
- **AC1**：不應再出現 `TypeError: string indices must be integers, not 'str'` 這類因回傳結構不符導致的執行緒崩潰。
- **AC2**：若 Gemini 回傳格式錯誤導致 JSONDecodeError，`word_to_card.log` 可看到該次回應的原始文字內容（至少包含錯誤附近片段）。

## 需求：單字去重（本地歷史 / 同批 / 佇列重試）
- **正規化規則**：去重比對時需對 `word` 做正規化（`strip` 去除前後空白、`lower` 忽略大小寫，並將連續空白視為同一個空白）。
- **同批去重**：Gemini 若在同一次回傳中給了重複的 `word`，只保留第一筆，其餘忽略。
- **歷史去重**：所有要送進 Anki 的單字（包含快捷鍵即時流程與佇列重試流程）都必須先經過歷史過濾，避免重複新增。
- **Anki 回饋對齊**：若 AnkiConnect 回報部分筆數為 duplicate（`addNotes` 回傳 `None`），只把實際成功新增的那幾筆寫入歷史，避免歷史與 Anki 不一致。

### 驗收條件
- **AC1**：啟動時重試佇列（`process_queue()`）不得再出現「同一單字重複送入 Anki」造成的 duplicate 錯誤；遇到重複單字時應在送出前就被過濾掉。
- **AC2**：同一次 Gemini 回傳內含重複單字時，寫入 Anki 的張數不得因重複而增加。
- **AC3**：佇列重試時，若去重後沒有任何新單字（代表全部重複/已收錄），該任務必須視為完成並從佇列移除，同時刪除截圖檔案。

## 需求：自動加入發音音檔（TTS → Anki Audio 欄位）
### 目標
- 在建立單字卡時，自動為每個單字產生英文發音 `.mp3`，並透過 AnkiConnect 寫入卡片的 **`Audio`** 欄位（媒體檔），同時在 **`Front` 欄位**附上可播放的 `[sound:...]`（**難度**置於音檔**下方**，見「Back／Front 版面」）。

### 流程
- **Step 1**：呼叫 Gemini（Vision/Text）取得單字卡 JSON（包含 `word`）。
- **Step 2**：對每個要新增的單字執行 TTS：
  - 輸入：單字（英文）
  - 輸出：`.mp3` 音檔（檔名需可重現且避免衝突，例如 `tts_<normalized_word>.mp3`，或加上短時間戳/雜湊）
- **Step 3**：寫入 Anki：
  - 使用 AnkiConnect 的 `audio` 屬性（例如 `addNote/addNotes` 的 `audio` 欄位）
  - 將音檔加入 Anki 媒體庫，並把播放標記寫進卡片的 `Audio` 欄位
  - **同時**在卡片的 `Front` 欄位附上 `[sound:filename.mp3]`（`difficulty` 顯示於該音檔**下方**，見「Back／Front 版面」）
- **Step 4（Cleanup）**：任務結束後，必須刪除本地暫存的 `.mp3` 檔案（不應累積在專案目錄）。

### 套件與選型
- **推薦**：`edge-tts`（語音自然度佳、可選 voice；需可連網）

### 錯誤處理
- 若 TTS 失敗：
  - 不應阻止整張卡片建立（可選策略：仍建立卡片但 Audio 留空，並記錄/通知 TTS 失敗原因）。
- 若 AnkiConnect 音檔寫入失敗：
  - 視為 Anki 寫入失敗同等級處理（應可進入佇列重試/或至少讓使用者知曉）。

### 驗收條件
- **AC1**：新增單字卡後，卡片的 `Audio` 欄位可播放該單字的英文發音（Anki 媒體庫中存在對應 `.mp3`）。
- **AC1.1**：新增單字卡後，卡片的 `Front` 欄位可看到可播放音檔（`[sound:...]`），且**難度標籤**顯示於該音檔**下方**。
- **AC2**：音檔產生後，本地暫存檔案會在成功/失敗後被清理，不會在專案目錄累積。
- **AC3**：當 TTS 服務暫時失敗時，流程仍能完成建立卡片（Audio 可為空），並在 log 中記錄失敗原因。
- **AC4**：同一欄位不得重複插入相同的 sound tag（例如 `Audio` 欄位不可出現 `[sound:x.mp3][sound:x.mp3]`）。

## 需求：Back／Front 版面（多義項區塊、`Front` 音檔與難度）
- **資料模型**：以 **`senses`**（陣列）描述 1～多個義項（詳見「多義項」需求，含**保守拆分原則**）；每筆含 `part_of_speech`、英／中定義、選填 Synonyms、選填 Usage、以及**該義項專用**的 `example_sentence`（1～2 句，`\n` 分行）。`difficulty` 與 `phonetic` 仍在根層（**`Back` 顯示順序**見下條「單一義項區塊內順序」）。
- **`Front` 由上而下**：單字 → 音標 → **音檔**（整卡僅一處 `[sound:...]`）→ **`difficulty`**。**不**在 `Front` 顯示詞性（詞性僅出現在 `Back` 各義項與通知預覽等）。
- **`Back` 由上而下**：僅含義項內容，不含難度與音檔。**每一義項區塊之間**須以水平線（`<hr>` 或等效樣式）隔開；多義項時**不**顯示「Other senses:」等額外標題，僅以分隔線區隔各義項。
- **單一義項區塊內順序**：英文定義 → **同一行**：`part_of_speech`（詞性）**在前** + 空格 + `definition_zh`（中文釋義，可含句末語體／場景括註），**整行**使用與原中文釋義相同的字級與顏色（例如 `verb 正式規定；命令…`）；僅無中文時才單獨顯示詞性該行 → Synonyms → Usage → Examples。
- **例句目標詞標示（紅字）**：
  - **是否可行**：可行；Anki `Back` 為 HTML，可對片段加 `<span style="color:…">`。
  - **是否建議**：建議由模型在 `example_sentence` 內使用**固定、罕見於英文正文**的括號標記目標詞，再由程式轉成紅色；**不建議**模型直接輸出任意 HTML（安全、樣式一致、易驗證）。
  - **格式（Gemini 輸出）**：以 Unicode **`⟦`**（U+27E6）與 **`⟧`**（U+27E7）包住句中的**目標單字或連續片語**（與該義項／`word` 對應）。**務必**包住句中**實際寫出**的連續字串（含**曲折變化**：過去式／過去分詞的 **-ed**、進行式 **-ing**、三人稱 **-s** 等皆須留在括號內，勿只標原形而漏掉詞尾）。**常用搭配／片語**若跨多字，應**整段**一併標示（同一對 `⟦⟧` 內），例如 *These factors alone do not fully account for the phenomenon.* 應標成 `⟦account for⟧`（不可只標 `account` 而漏 `for`）；若句中為過去式則標 `⟦accounted for⟧` 等**與原文完全一致**的片段。**每句最多一處**一對括號（若有第二句 `\n` 分隔，則第二句可再各標一處）。除上述括號外**不要**輸出其他 HTML。
  - **產卡行為**：程式將 `⟦…⟧` 轉為紅色顯示（其餘字元 HTML escape）；**無**括號的舊例句仍整句斜體顯示，行為與過去相同。
- **對齊**：`Back` 義項區塊內容靠左；`Front` 維持置中為主（音檔與難度亦置中）。
- **相容**：舊版根欄位 + `other_senses` 經正規化為 `senses` 後，與新版同一套渲染。

### 驗收條件
- **AC1**：`Front` 可見可播放音檔與其**下方**之難度；`Front` 不含詞性列；`Back` 不含難度與音檔。
- **AC2**：多義項時 `Back` 上**每兩個義項區塊之間**可見分隔線，且不出現「Other senses:」字樣。
- **AC3**：音檔僅出現於 `Front` 一次，不重複插入 `Back`。
- **AC4**（例句紅字）：`example_sentence` 中含成對 `⟦…⟧` 時，`Back` 上對應片段以紅色顯示；無括號的例句仍可正常閱讀。

## 需求：成功通知內容（單字 + 中文）
- 當卡片成功新增時，系統通知內容需同時包含：
  - 英文單字 `word`
  - 詞性縮寫（由 `part_of_speech` 轉換，例如 noun -> `n.`）
  - 中文意思（優先 `definition_zh`）
- 若一次新增多個單字，通知中每筆仍需保留「英文 + 詞性 + 中文」的對照格式（例如 `player (n.) 玩家`）。

### 驗收條件
- **AC1**：成功通知不只顯示英文單字，必須可看到對應中文意思。
- **AC2**：成功通知可看到詞性縮寫（如 `n.`、`v.`）。

## 需求：移除 Context 顯示
- 不再要求 Gemini 回傳 `context` 欄位。
- 卡片 `Back` 不顯示 `Context: ...` 區塊，避免 OCR 殘句造成雜訊。

### 驗收條件
- **AC1**：新建立卡片的 `Back` 不再出現 `Context:` 區塊。

## 需求：常用用法（含介系詞搭配）
- **適用範圍**：每個義項（`senses[]` 內或舊版根層級）各自可有 `usage_patterns`。
- **資料欄位**：`usage_patterns` 為字串陣列，內容為該義項**高訊號**的常見搭配/用法（例如 `attribute A to B`、`interested in`）。
- **數量原則**：每個義項內 **0～4 筆**；不要硬湊數量；無則 `[]`。
- **顯示格式**：`Usage` 需有標題，且每一筆搭配各自一行（清單格式）。

### 驗收條件
- **AC1**：某義項的 `usage_patterns` 非空時，該義項區塊內可看到對應 usage 清單。
- **AC2**：空陣列時該義項不顯示 Usage 區塊，卡片仍可建立。

## 需求：多義項（`senses` 陣列）
### 背景
- 同一拼字在少數情況下會有 **值得分開學的** 多個義項或詞性（例如 *Pacific* 形容詞「溫和的」與專有名詞「太平洋」），且該分歧須對 **TOEFL／IELTS、商用英文、C2** 等目標有實質幫助；此時學習者才需要在**不同區塊**各自看到定義、用法與**專屬例句**。

### 拆分原則（重要）
- **「足夠重要」的操作定義**：該分歧須**足以**在 **TOEFL／IELTS**、**商用英文**（職場溝通、簡報、書信）或 **CEFR C2** 等級的閱讀／聽力／寫作中**各自獨立出現**、且考生或進階學習者**若不拆開記就容易混淆或選錯義**者，才算值得拆成多區塊。若拆開後對上述場景幾乎沒有加分（仍屬同一學習單元即可掌握），則應合併為 **1 筆** `senses`。
- **預設單區塊**：除非符合上列「足夠重要」標準，否則 **`senses` 應只有 1 筆**（整張卡一個義項區塊即可）。
- **才拆多區塊**的情況（須同時滿足「TOEFL／IELTS／商用／C2 場景下值得分開記」且「語義或用法明顯分岔」），例如：專有名詞 vs 一般詞性、動詞 vs 名詞且核心語義不同、兩個義項各自需要不同例句與搭配才說得清楚等。
- **熟詞僻義（考試高頻）**：同一拼字若除**日常／直觀義**外，另有 **GRE／TOEFL 讀寫常考的次要但關鍵義**（熟詞僻義），且與前者在語義或句型上**易讀錯、選錯義**，應**辨識並分筆**列入 `senses`（各自定義、例句、搭配），**不可**只輸出日常義而略過僻義。**範例**：*dispose*——一義為「處置／清除」（如 *dispose of waste*）；另一義為「使（某人）傾向於／使易於（某態度或行為）」（如 *dispose someone to sympathy*、*well-disposed* 一脈語義）。兩義在考試與學術語境中皆須能分辨時，應拆成**至少兩筆** `senses`（仍受至多 4 筆與排序規則約束）。
- **不應拆分**的情況：細微語氣差、同一核心意義的換說法、可在一筆定義中用括註或簡短補句交代者、僅為同義替換或輕微延伸者——請合併在 **單一 `senses` 條目**內處理。
- **須捨棄的義項**：若某義項**僅見於古典文學／詩歌**或**極罕見**現代語境，且**不屬於 TOEFL／GRE 常考或一般學術／現代英語常用範圍**，**不得**列入 `senses`（直接省略），以降低使用者記憶負擔。

### 資料欄位（Gemini，優先）
- **`senses`**（陣列，必填）：至少 1 筆、建議至多 **4** 筆；**多數單字應為 1 筆**。僅在符合上列「拆分原則」時才增加第二筆以後。
- **排序（必填）**：`senses` 內各筆必須依**對 TOEFL／IELTS、商用英文、CEFR C2 的重要性與實際使用頻率**由**高到低**排列（`senses[0]` 為最重要、最常用或最符合截圖／輸入語境者；其後遞減）。**不得**把冷僻義放在前面。
- 每筆物件含：
  - **`part_of_speech`（必填）**：非動詞類沿用慣用英文（如 `noun`、`adjective`、`proper noun`）。**動詞**（含一般動詞與片語動詞）：**必須**標明及物性，**不得**僅輸出 `verb`；格式為 **`verb (vt)`**、**`verb (vi)`** 或 **`verb (vi/vt)`**（括號內表及物／不及物／兼用；兼用時建議寫成 `vi/vt`）。片語動詞則為 **`phrasal verb (vt)`**、**`phrasal verb (vi)`**、**`phrasal verb (vi/vt)`**（全小寫，括號前後空格可省略但建議 `verb (vi)` 有一空格）。
  - `definition`、`definition_zh`（必填，精簡）
  - **`definition_zh` 語體／場景括註（選填）**：當該義項在**書面 vs 口語**、**正式 vs 非正式**、**學術 vs 日常／商務**等維度上有助學習者判斷「此義多常出現於何類情境」時，可在中文釋義**結尾**以括號簡短加註（例如「（書面／學術較常）」「（口語常用）」「（正式場合較常）」）。若各維度差異不大、加註無增益或會變冗長，則**省略**。
  - `synonyms`：0～3（選填）
  - `usage_patterns`：0～4（選填，規則同「常用用法」）
  - `example_sentence`：**1～2 句**英文例句，**僅能對應本義項**，建議以 `\n` 分隔；**必須**依「Back／Front 版面」之 **`⟦`…`⟧` 目標詞標示**規則標出目標詞或**完整連續片語**（含 **-ed／-ing／-s** 等句中實際字形；片語如 *account for*、*accounted for* 須整段一對括號內標完），每句最多一處一對括號，**不要**輸出其他 HTML
- **卡片層級**：`word`、`phonetic`、`difficulty` 仍置於物件根層（整卡共用）。
- **相容舊版**：若模型未回傳 `senses`，仍可回傳根層 `part_of_speech`、`definition`、`definition_zh`、`example_sentence`、`synonyms`、`usage_patterns` 加上選填 `other_senses`（每筆可含 `example_sentence`、`synonyms`）；程式正規化為 `senses` 後再送 Anki。

### 驗收條件
- **AC1**：`senses` 含 2 筆以上時，`Back` 各義項僅以分隔線區隔，且每筆皆有獨立 **Examples:**。由 Gemini 新產出之例句應依「Back／Front 版面」以 `⟦…⟧` 標示目標詞；無標示之舊資料仍可正常顯示。
- **AC2**：`senses` 僅 1 筆時為單一義項區塊，版面不出現多義項分隔線群。
- **AC3**：舊版 JSON 僅有根欄位 + `other_senses` 時，仍能產生與新模型類似的多區塊 `Back`（在欄位足夠的前提下）。
- **AC4**（行為準則）：模型在「可合併為一義項」時應優先輸出 **1 筆** `senses`；避免為細微差別硬拆多區塊；多區塊僅在符合 **TOEFL／IELTS／商用英文／C2** 情境下「值得分開記」時使用。
- **AC5**：多義項時，`senses` 順序須符合「重要度／常用度由高到低」；必要時 `definition_zh` 可含語體／場景括註以利辨識（見「資料欄位」）。
- **AC6**：模型不得輸出「僅古典／極罕見且非 TOEFL／GRE 常考」的義項至 `senses`（應直接捨棄）。
- **AC7**：凡 `senses` 中詞性為**動詞**（含片語動詞）者，`part_of_speech` 須符合上列 **`verb (vt|vi|vi/vt)`** 括號格式（片語則 `phrasal verb (...)`）；通知預覽之詞性縮寫應能對應顯示為 `vt.`／`vi.`／`vi./vt.`（片語加 `phr.`）。
- **AC8**（熟詞僻義）：當單字同時具備「明顯日常義」與「GRE／TOEFL 常考僻義」且符合拆分原則時，模型輸出之 `senses` 須**涵蓋該僻義**（獨立一筆或與其他考點分筆），且排序須以**畫面／輸入語境與考試重要度**為準；**不得**僅剩單一日常釋義而遺漏常考僻義（在筆數上限內）。

## 需求：本地單字資料目錄（與 Anki 並行）
### 目標
- 除了 Anki 牌組外，使用者在專案內還有一份**可自行維護**的單字資料（檔案系統目錄），便於備份、搜尋或日後匯出。

### 行為
- **觸發時機**：僅在 AnkiConnect **確認新增成功**的單字上執行（與 `history_logger.record` 使用同一批 `added_words`）；若該批全部為 duplicate（無成功筆數），則不寫入本地檔案。
- **儲存位置**：預設為專案根目錄下的 `vocabulary/`；可透過環境變數 `WORD_ARCHIVE_DIR` 覆寫（相對路徑則相對於專案根目錄）。
- **停用**：可透過 `WORD_ARCHIVE_ENABLED=false`（或 `0`）關閉此功能。
- **檔案格式**：每個單字一個 `*.json` 檔；**僅**寫入 `word`、`phonetic`、`difficulty`、`senses` 與 **`archived_at`**（ISO 8601 UTC）。**不**寫入與 `senses[0]` 重複的根層鏡像（如 `part_of_speech`、`definition_zh`、`other_senses` 等）；執行流程記憶體內仍可保留鏡像供 Anki／通知使用。
- **檔名規則**：以與歷史去重相同的 **`word` 正規化**（小寫、空白正規化）產生安全檔名；同一單字再次成功存檔時**覆寫**同一檔案（以最新內容為準）。
- **錯誤隔離**：若目錄無法建立或單檔寫入失敗，不應影響 Anki 寫入與歷史紀錄；僅記錄 log 警告。

### 驗收條件
- **AC1**：成功新增至少一張 Anki 卡片後，專案預設的 `vocabulary/` 下可看到對應的 `.json` 檔，且內容含 `word`、`senses` 與 `archived_at`（精簡格式，無根層鏡像重複欄位）。
- **AC2**：`WORD_ARCHIVE_ENABLED=false` 時，不建立或更新 `vocabulary/`（或指定目錄）中的檔案。
- **AC3**：佇列重試成功新增時，行為與即時流程一致（成功筆數仍會寫入本地）。
